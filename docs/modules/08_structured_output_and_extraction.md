# Module 8 — Structured Output and Extraction

> Phase: Application primitives · Bible reference: [curriculum.md §18](../../curriculum.md#18-module-8--structured-output-and-extraction)

## Goal

Build reliable local extraction systems — the full production pipeline, with constrained
decoding as the primary reliability layer rather than a retry loop bolted onto "please
return JSON."

> **Machine note:** this repo is built on a Mac that must never run a model runtime
> ([[project-local-ai-app-curriculum]] constraint; target execution hardware confirmed as a
> separate 32GB Mac). Like Modules 6/6.5/7, the pipeline itself is fully built and tested
> against `FakeRuntime`. The three-way constrained-decoding comparison (Lab 8) is
> demonstrated with a fake configured to exhibit realistic reliability differences, clearly
> labeled as such; the real comparison is honest-skip pending the resourced Mac.

## 1. Why free-form output is fragile

A model asked to "return JSON" in prose is free to also return markdown fences, commentary
before or after the JSON, or a subtly malformed structure — none of which violate the
prompt's *intent*, because prompt-level instructions are suggestions the model is trained to
follow probabilistically, not a hard constraint on the token stream. Module 1 §11 already
named this pattern ("format collapse under complexity"); this module builds the fix.

## 2. JSON mode vs. schema-constrained output

"JSON mode" (a runtime flag that biases toward *some* valid JSON) and full schema-constrained
decoding (the output must match a specific JSON Schema, key-for-key) are different strengths
of guarantee. Module 6's `ResponseFormat.type` already distinguishes `"json_schema"` from
free text; this module is where an application actually uses that distinction instead of
just declaring it.

## 3-4. Pydantic schemas and validation

`packages/local_ai_core/extraction/schemas.py` defines the schemas this module's pipeline
validates against — Pydantic models double as the JSON Schema source
(`Model.model_json_schema()`) *and* the validator, so the schema sent to the runtime for
constrained decoding and the schema used to validate the result are guaranteed to be the
same schema, not two hand-maintained copies that can drift apart.

## 5. Retry and repair

A **repair prompt** shows the model its own invalid output and the specific validation error,
and asks it to fix exactly that — a narrower, more effective ask than re-running the original
prompt from scratch and hoping for a better roll. `pipeline.py`'s `build_repair_prompt()`
implements this. Per the Gotchas below, repair retries only happen a bounded number of times
and only for genuinely fixable failures, since retrying can amplify hallucination if the
underlying problem is that the model doesn't know the answer, not that it phrased it wrong.

## 6. Partial extraction

Long documents may need chunking (Module 1 §6 context budget); `chunking.py`'s
`chunk_text()` splits by paragraph with configurable overlap, and
`merge_partial_extractions()` combines per-chunk results — preferring the first non-null
value found for each field and flagging fields where chunks disagree, rather than silently
picking one.

## 7. Confidence scoring

**Confidence generated by the model is not trustworthy by itself** (Gotcha, verbatim) — a
model asked to self-report confidence is still just generating plausible-looking text, with
no calibration guarantee. `confidence.py`'s `compute_confidence()` is deliberately
**deterministic and external to the model**: it scores based on measurable pipeline facts
(how many required fields came back non-null, whether a repair retry was needed, whether
constrained decoding was actually used) rather than trusting whatever the model wrote in a
`confidence` field of its own output.

## 8. Human review queues

Low-confidence extractions get queued for human review rather than silently persisted or
silently discarded — `review_queue.py`'s `ReviewQueue` is the last resort in the reliability
ladder (§"Constrained decoding is the primary reliability layer" below), for records where
every automated layer has already been applied and the result still isn't trustworthy enough
to use unreviewed.

## 9. Schema evolution

A schema will change over time (a new field, a renamed field, a stricter constraint). Module
6.5's `response_cache_key()` already takes a `schema_version` parameter for exactly this
reason — a schema change must invalidate cached extractions keyed to the old schema, not
silently keep serving stale-shaped cached results.

## 10. Golden test sets

`evals/golden_sets/extraction.jsonl` (built in Module 3) is reused here rather than
duplicated — the same frozen extraction records that benchmarked model selection now
evaluate this module's pipeline against known-correct field values (Lab 7).

## Production extraction pipeline

```text
Input document
  -> normalization
  -> chunking if needed
  -> prompt assembly
  -> local LLM call with constrained decoding when supported
  -> parse JSON
  -> validate schema
  -> repair retry if needed
  -> deterministic checks
  -> confidence scoring
  -> persist result
  -> human review if low confidence
```

`ExtractionPipeline.run()` in `pipeline.py` implements every stage except "persist result" —
persistence is application-specific (SQLite, a database, a file) and out of scope for a
reusable pipeline component; the pipeline returns a complete, typed result the caller decides
how to store.

## Constrained decoding is the primary reliability layer

```text
1. Constrained decoding by grammar or JSON schema  # strongest structural guarantee
2. Schema validation with Pydantic                 # catches semantic and business-rule issues
3. Retry with repair prompt                        # fallback for semantic failures or unsupported runtime features
4. Human review queue                              # last resort for low-confidence or high-risk records
```

The beginner mistake is starting with "please return JSON" and building retry loops around
it. `ExtractionPipeline` always requests `response_format.type="json_schema"` first — Module
6's adapters either honor it (Ollama, OpenAI-compatible) or raise `FeatureNotSupported`
(caught and logged, falling back to prompt-only mode with a recorded reason), never silently
degrading without telling the caller which mode was actually used.

### What constrained decoding does

At each decode step, the runtime masks token choices so the model can only emit tokens
allowed by a grammar or schema. The model cannot produce structurally invalid output,
although it can still produce wrong content — `"invoice_number": "N/A"` is valid JSON and a
useless answer at the same time. Structural guarantee and semantic correctness are different
properties, and this module's confidence scoring (§7) is partly aimed at the gap between them.

| Approach | Where to teach it | Notes |
|---|---|---|
| GBNF grammar | llama.cpp / llama-cpp-python style runtimes | Most control; useful for enums, constrained shapes, non-JSON formats |
| JSON-schema mode | Ollama-style structured output, llama.cpp-compatible APIs | Ergonomic; uses `PydanticModel.model_json_schema()` directly |
| `outlines` | offline pipelines with Transformers/llama.cpp integrations | Regex, JSON Schema, and grammar-oriented generation |
| `lm-format-enforcer` | token-level format enforcement | Useful when runtime-native grammar support is missing |
| `xgrammar` | high-performance structured-output engines | Ecosystem-direction awareness, not used in this module's labs |

### Critical gotchas

- Constrained decoding guarantees valid shape, not correct content.
- Over-tight grammars can hurt quality and latency — Lab 8 compares constrained and
  unconstrained variants on the same golden set specifically to measure this, not assume it.
- Runtime support differs — Module 6's `response_format` field translates to runtime-specific
  behavior or raises `FeatureNotSupported`; this module's pipeline treats that exception as a
  signal to fall back, not a crash.
- Streaming conflicts with structured output (§"Streaming vs structured output" below).

## Validation strategy

```text
1. Runtime-level constrained decoding where supported.
2. JSON parse validation.
3. Pydantic schema validation.
4. Business rule validation.
5. Cross-field validation.
6. Confidence scoring.
7. Human review.
```

`ExtractionPipeline.run()` runs every layer in order, short-circuiting to repair-retry only
on layers 2-4's failures (a parse or schema failure is potentially fixable by asking the
model to correct it; layers 6-7 are about the *result* of a structurally-valid extraction,
not something a repair prompt can address).

## Streaming vs structured output

**Rule of thumb: stream prose, buffer structure.** `ExtractionPipeline` exclusively calls
`LLMRuntime.generate()` (buffered), never `.stream()` — a half-streamed JSON object is not
authoritative and extraction results feed deterministic downstream code (this pipeline
itself, then whatever calls it), which is exactly the case Module 6's theory doc flagged as
needing buffering, not streaming.

## Example schema

```python
class InvoiceExtraction(BaseModel):
    invoice_number: str | None = None
    vendor_name: str | None = None
    invoice_date: str | None = Field(default=None, description="ISO date if present")
    currency: str | None = None
    total_amount: float | None = None
    confidence: Literal["low", "medium", "high"]
    evidence: dict[str, str] = Field(default_factory=dict)
```

Implemented verbatim in `schemas.py`, alongside a `PersonExtraction` schema matching the
name/age/city task Modules 1, 3, and 7 have already used as a running example — reusing the
same example task across modules rather than introducing a new one for its own sake.

Note on the schema's own `confidence` field: the pipeline does not populate it from the
model's self-report (§7's Gotcha) — it's included in the schema to match curriculum's example
and can hold the pipeline's own `compute_confidence()` result if a caller chooses to fold it
back in, but the pipeline's authoritative confidence lives in `ExtractionResult.confidence`,
computed deterministically outside the model's output.

## Gotchas

- Models may return markdown-wrapped JSON — handled by `json_parsing.py`'s own
  `strip_markdown_fence()`, a small intentional duplication of Module 3's
  `scorers/json_validity.py` logic rather than a cross-boundary import: `packages/` must not
  depend on `scripts/module_NN/` (the reverse is expected and common), so a ~15-line helper
  is duplicated here rather than inverting that layering for one function.
- Models may add comments inside JSON.
- Models may invent missing fields — cross-field/business-rule validation (layer 4) is where
  this gets caught, not schema validation alone (a hallucinated but well-typed value passes
  schema validation trivially).
- Small models may follow examples more than rules — Module 7's few-shot infrastructure is
  reused in the repair prompt for exactly this reason.
- Retry can amplify hallucination if not constrained — repair retries are capped (default 1)
  and only triggered by structural/schema failures, never by low confidence alone.
- Schema validation is necessary but not sufficient (§ example schema note above).
- Confidence generated by the model is not trustworthy by itself (§7).

1. **Extract structured data from short text** — `ExtractionPipeline.run()` against a single
   short input.
