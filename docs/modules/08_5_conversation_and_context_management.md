# Module 8.5 — Conversation and Context Management

> Phase: Application primitives · Bible reference: [curriculum.md §18.5](../../curriculum.md#185-module-85--conversation-and-context-management)

## Goal

Manage multi-turn conversation state within a small local context window without silent
truncation, broken chat templates, or lost task state.

> **Machine note:** this repo is built on a Mac that must never run a model runtime
> ([[project-local-ai-app-curriculum]] constraint; target execution hardware confirmed as a
> separate 32GB Mac). Like every module since Module 6, the conversation infrastructure
> itself is fully built and tested without a live runtime — `SessionStore` uses Python's
> stdlib `sqlite3` directly (no server, no external dependency), and every budget/truncation/
> summarization function accepts an injected token counter or summarize function. Only
> Lab 5 (recall measurement against a real model) is honest-skip.

## 1. Turn structure and model-specific chat templates

A conversation is a sequence of `Turn`s (`system`/`user`/`assistant`/`tool` roles), not a flat
string. Rendering that sequence into the actual prompt text a model receives is the chat
template's job — and it's model-specific (Module 1 §5, Module 6's `MLXRuntime.render_prompt`
already does this via `tokenizer.apply_chat_template`). This module manages the turn
sequence; rendering it is delegated to whichever adapter ultimately calls the model, not
duplicated here.

## 2. Token-aware history accounting

**Count tokens after rendering the full chat template, not before** (Gotcha, verbatim —
restating Module 1 §5's rule in the conversation context specifically, since role markers and
separators the chat template adds are exactly what a raw word-count would miss).
`token_budget.py`'s functions accept an injected `token_counter` callable
(`list[Turn] -> int`) rather than counting words themselves — real usage plugs in Module 1's
`HFTokenizerCounter.count_chat()` (which does apply the real chat template); this module's
own default is an explicitly-labeled heuristic, never trusted for a real budget decision, the
same discipline Module 1 established.

## 3-5. Drop-oldest, keep-system+last-N, importance-weighted retention

| Strategy | Works well for | Failure mode |
|---|---|---|
| Drop oldest | short transactional chats | loses early commitments/preferences |
| Last-N turns | support/chat UX | misses long-running task facts |
| Summarization buffer | long conversations | summary can lose important details |
| Importance-weighted retention | task/project conversations | needs scoring and traceability |
| RAG-backed memory | knowledge-heavy sessions | retrieval mistakes become memory mistakes |

`truncation.py` implements the first two directly (`drop_oldest`, `keep_system_plus_last_n`).
Importance-weighted retention and RAG-backed memory are documented here as the strategies
they are, not implemented — importance scoring needs a scoring function this module doesn't
own (it would be task-specific), and RAG-backed memory is Module 11's entire subject, not a
few functions bolted onto a conversation module.

## 6. Summarization buffer

`summarizer.py`'s `summarize_then_truncate()` replaces older turns with a single summary
turn, generated by an injected `summarize_fn` (the same dependency-injection principle as
every adapter since Module 6 — real usage wraps `LLMRuntime.generate()`, tests inject a
fake). Per the Gotcha below, it always keeps the last 1-2 raw turns even when summarizing
everything before them.

## 7. Sticky context

System prompt, tools, current task, and safety policy stay pinned regardless of which
truncation strategy runs. `Turn.sticky: bool` marks a turn as exempt from every truncation
strategy in this module — `drop_oldest` and `keep_system_plus_last_n` both check it before
considering a turn for removal.

## 8-9. Session persistence in SQLite and resumption after restart

`session_store.py`'s `SessionStore` persists every turn to a SQLite database via the stdlib
`sqlite3` module — no server, no new dependency. Resumption is proven directly: a test opens
a store, appends turns, closes the connection, reopens a **new** `SessionStore` instance
against the same file path, and confirms every turn is still there — not just that
"persistence" is implemented, but that it survives an actual close/reopen cycle.

## 10. Separation of conversation memory, retrieved RAG memory, and tool state

**Retrieved context is not conversation memory. Tool state is not conversation memory. Keep
them separate in storage and traces** (Gotcha, verbatim). `SessionStore`'s schema only has
columns for conversation turns (role, content, tool linkage, timestamps) — there is
deliberately no column for retrieved-document content or arbitrary application state. This
absence *is* the enforcement: a caller that wants to store retrieved context has nowhere to
put it in this schema and must use a separate store (Module 11's job), rather than the
schema quietly accommodating scope creep.

## 11. Tool-call/tool-result turn pairing

A tool-call (assistant turn proposing a call) and its tool-result (the tool's response turn)
must be kept or dropped **as a unit** — truncating between them leaves a dangling tool call
with no result, which breaks the conversation's validity for most chat template formats.
`truncation.py`'s `group_turns()` groups a tool-call turn with its paired result turn (linked
via `Turn.turn_group_id`) into one atomic unit before either truncation strategy runs, so
neither strategy can split a pair.

## 12. Memory privacy and deletion

`SessionStore.delete_session()` removes every turn for a session — Lab 6's "memory deletion
command" is this method exposed at the CLI/application layer, not new logic.

## Conversation budget

```text
history_budget = context_window
               - reserved_system
               - reserved_tools
               - reserved_current_user_turn
               - reserved_answer
```

`ConversationBudget.history_budget` computes exactly this, clamped to zero (never negative —
a misconfigured budget should read as "no room for history," not silently underflow).

When `tokens(history) > history_budget`, summarize-then-truncate. **Never hard-cut in the
middle of a user, assistant, or tool-result turn** — every function in this module operates
on whole `Turn`s (or whole tool-call/result groups), never on substrings of a turn's content.

## Gotchas

- Chat templates are model-specific. Using the wrong template silently degrades quality —
  this module never renders a template itself, precisely to avoid this failure mode; it
  manages turns, rendering is the adapter's job (§1).
- Count tokens after rendering the full chat template, not before (§2).
- Summarization is lossy; keep the last one or two raw turns even when old turns are
  summarized (§6, `summarize_then_truncate`'s `keep_last_n_raw` parameter, default 2).
- Tool-call and tool-result turns must be kept as a unit (§11, `group_turns()`).
- Retrieved context is not conversation memory. Tool state is not conversation memory. Keep
  them separate in storage and traces (§10, enforced by `SessionStore`'s schema).
